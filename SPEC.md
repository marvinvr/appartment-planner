# FloorPlanner – Full Product Specification

## Overview

A self-hosted, single-container web application for planning apartment furniture layouts.
Upload a floor plan PDF once, calibrate the scale against a known real-world distance,
build a furniture library with real-world dimensions, then create and compare multiple
layout variants of the same floor plan – all in the browser, persisted in a local SQL
database.

## Database Schema (`src/db/schema.ts`)

```ts
// projects -- one per apartment / floor plan upload
export const projects = sqliteTable("projects", {
  id:              integer("id").primaryKey({ autoIncrement: true }),
  name:            text("name").notNull(),
  pdfPath:         text("pdf_path"),                  // relative path under /data/uploads/
  pdfWidthPx:      integer("pdf_width_px"),           // rendered page width in px (stored after first render signal from client)
  pdfHeightPx:     integer("pdf_height_px"),          // rendered page height in px
  // Calibration: two pixel coordinates of the drawn reference line
  calX1:           real("cal_x1"),
  calY1:           real("cal_y1"),
  calX2:           real("cal_x2"),
  calY2:           real("cal_y2"),
  calRealCm:       real("cal_real_cm"),               // user-entered real-world length in cm
  pixelsPerCm:     real("pixels_per_cm"),             // derived: pixel_distance / calRealCm
  createdAt:       integer("created_at", { mode: "timestamp" }).notNull().$defaultFn(() => new Date()),
});

// furniture_items -- the shared library for a project
export const furnitureItems = sqliteTable("furniture_items", {
  id:         integer("id").primaryKey({ autoIncrement: true }),
  projectId:  integer("project_id").notNull().references(() => projects.id, { onDelete: "cascade" }),
  name:       text("name").notNull(),
  widthCm:    real("width_cm").notNull(),
  heightCm:   real("height_cm").notNull(),
  color:      text("color").notNull().default("#93c5fd"),  // tailwind-style hex default
  createdAt:  integer("created_at", { mode: "timestamp" }).notNull().$defaultFn(() => new Date()),
});

// layouts -- named variants of the floor plan
export const layouts = sqliteTable("layouts", {
  id:         integer("id").primaryKey({ autoIncrement: true }),
  projectId:  integer("project_id").notNull().references(() => projects.id, { onDelete: "cascade" }),
  name:       text("name").notNull(),
  createdAt:  integer("created_at", { mode: "timestamp" }).notNull().$defaultFn(() => new Date()),
});

// placements -- one furniture item placed in one layout at a position
export const placements = sqliteTable("placements", {
  id:              integer("id").primaryKey({ autoIncrement: true }),
  layoutId:        integer("layout_id").notNull().references(() => layouts.id, { onDelete: "cascade" }),
  furnitureItemId: integer("furniture_item_id").notNull().references(() => furnitureItems.id, { onDelete: "cascade" }),
  xPx:             real("x_px").notNull(),        // canvas pixel X of top-left corner
  yPx:             real("y_px").notNull(),        // canvas pixel Y of top-left corner
  rotationDeg:     real("rotation_deg").notNull().default(0),
  updatedAt:       integer("updated_at", { mode: "timestamp" }).notNull().$defaultFn(() => new Date()),
});
```

**Notes:**

- All furniture dimensions are stored and communicated in **centimetres**.
- Canvas positions are stored in **canvas pixels** (i.e., the coordinate space of the rendered
  PDF image on the Konva stage). This makes them stable regardless of display zoom/DPR.
- Scaling from cm to canvas-px at render time: `widthPx = widthCm * pixelsPerCm`.

## API Routes

All routes return `application/json`. Errors return `{ error: string }` with an appropriate
HTTP status code.

### Projects

```
GET    /api/projects                       List all projects
POST   /api/projects                       Create project { name: string }
GET    /api/projects/[projectId]           Get single project (includes calibration fields)
DELETE /api/projects/[projectId]           Delete project + cascade all data + delete PDF file
```

### PDF

```
POST   /api/projects/[projectId]/pdf       Upload PDF (multipart/form-data, field: "file")
                                            Saves to /data/uploads/<projectId>.pdf
                                            Returns { pdfUrl: string }
GET    /api/pdf/[projectId]                Serve raw PDF bytes (Content-Type: application/pdf)
                                            Used by pdfjs-dist on the client
```

PDF upload endpoint must also accept a body field `{ pdfWidthPx, pdfHeightPx }` on a
subsequent `PATCH /api/projects/[projectId]` call once the client has rendered the first
page and knows its pixel dimensions.

### Calibration

```
POST   /api/projects/[projectId]/calibration
  Body: { x1: number, y1: number, x2: number, y2: number, realCm: number }
  Computes pixelsPerCm = sqrt((x2-x1)^2 + (y2-y1)^2) / realCm
  Saves all calibration fields to projects row
  Returns updated project
```

### Furniture Library

```
GET    /api/projects/[projectId]/furniture  List all furniture items for this project
POST   /api/projects/[projectId]/furniture  Create item { name, widthCm, heightCm, color }
PUT    /api/furniture/[furnitureId]          Update item (same fields, all optional)
DELETE /api/furniture/[furnitureId]          Delete item (also removes all placements of it)
```

### Layouts

```
GET    /api/projects/[projectId]/layouts    List layouts for project
POST   /api/projects/[projectId]/layouts    Create layout { name: string }
GET    /api/layouts/[layoutId]              Get layout + all its placements (joined with furniture item data)
PUT    /api/layouts/[layoutId]             Rename layout { name: string }
DELETE /api/layouts/[layoutId]             Delete layout + all placements
```

### Placements

```
POST   /api/layouts/[layoutId]/placements
  Body: { furnitureItemId, xPx, yPx, rotationDeg? }
  Returns created placement

PUT    /api/layouts/[layoutId]/placements/[placementId]
  Body: { xPx?, yPx?, rotationDeg? }
  Called on every drag-end event (debounced 200ms)
  Returns updated placement

DELETE /api/layouts/[layoutId]/placements/[placementId]
  Removes a placed item from the layout
```

## User Flows

### Flow 1: Create a New Project

1. User visits `/` – sees list of existing projects + “New Project” button.
1. Clicks “New Project”, enters apartment name (e.g. “Gartenstrasse 12”).
1. POST `/api/projects` – redirected to `/projects/[id]/setup`.

### Flow 2: PDF Upload

On `/projects/[id]/setup`, first sub-step:

1. “Upload Floor Plan” area (drag-and-drop or file picker, PDF only).
1. Client POSTs to `/api/projects/[id]/pdf` with the file.
1. On success, the client renders page 1 of the PDF using `pdfjs-dist` into a hidden
   `<canvas>` element to determine the natural pixel dimensions of the rendered page.
1. Client PATCHes the project with `{ pdfWidthPx, pdfHeightPx }`.
1. UI advances to the calibration sub-step (same page, wizard-style).

### Flow 3: Calibration

Still on `/projects/[id]/setup`, second sub-step:

1. The rendered PDF is shown on a Konva Stage (read-only for now).
1. Instructions: “Click two points on the floor plan that correspond to a known distance.”
1. User clicks point A – a circle is drawn.
1. User clicks point B – a line is drawn between A and B.
1. A modal appears: “How long is this distance in real life? (cm)”.
1. User enters value, confirms.
1. Client POSTs to `/api/projects/[id]/calibration` with both pixel coords and the cm value.
1. On success, user is redirected to `/projects/[id]/furniture`.

If the project is already calibrated, the setup page shows current calibration and a
“Re-calibrate” button that restarts the flow.

### Flow 4: Furniture Library

On `/projects/[id]/furniture`:

1. Shows list of existing furniture items as cards (name, dimensions in cm, colour swatch).
1. “Add Item” button opens a form:
- Name (text)
- Width (cm, number)
- Height (cm, number)
- Colour (colour picker, hex)
1. Items can be edited inline or deleted.
1. “Go to Layouts” button navigates to `/projects/[id]/layouts`.

### Flow 5: Layouts

On `/projects/[id]/layouts`:

1. List of layout variants as cards (name, creation date).
1. “New Layout” button – prompts for a name, POSTs to API, redirects to editor.
1. Clicking an existing layout opens the editor.

### Flow 6: Canvas Editor

On `/projects/[id]/layouts/[layoutId]`:

```
+-------------------------------------------------------------+
|  [Project Name]  >  [Layout Name]  [Rename]  [Back]        |
+--------------------+----------------------------------------+
|                    |                                        |
|  FURNITURE         |                                        |
|  LIBRARY           |         KONVA CANVAS                   |
|                    |   (floor plan PDF as background)       |
|  [Sofa]            |                                        |
|  200 x 90 cm       |   [placed furniture rects]             |
|                    |                                        |
|  [Dining Table]    |                                        |
|  160 x 80 cm       |                                        |
|                    |                                        |
|  [Bed]             |                                        |
|  200 x 160 cm      |                                        |
|                    |                                        |
|  [+ Add Item]      |                                        |
|                    |                                        |
+--------------------+----------------------------------------+
|  Zoom: [--] [100%] [++]   |  Selected: Sofa  [Rotate 90°]  [Remove]  |
+-------------------------------------------------------------+
```

**Canvas behaviour:**

- The Konva Stage is sized to fit the container with CSS; the content is scaled via
  `Stage.scale()` for zoom. Internal coordinates are always in the natural PDF pixel space.
- The floor plan PDF page is rendered once client-side via pdfjs-dist into an `<img>` tag
  (or HTMLCanvasElement converted to data URL), then used as a Konva `Image` node as the
  background layer. This layer is non-interactive.
- Each placement is a Konva `Group` containing:
  - A `Rect` (fill = item colour, stroke = darker shade, opacity 0.75)
  - A `Text` node centered in the rect showing the item name
  - Dimensions are: `widthPx = item.widthCm * pixelsPerCm`, `heightPx = item.heightCm * pixelsPerCm`
- Placements are on a separate interactive layer above the background layer.
- Dragging a piece triggers `dragend` – calls PUT placement API (debounced).
- Clicking a piece selects it: shows a Konva `Transformer` for visual selection + enables
  the toolbar actions (rotate, remove).
- “Rotate 90°” button updates the placement’s `rotationDeg` by +90.
- “Remove” button calls DELETE placement API and removes the node.

**Adding furniture to canvas from sidebar:**

- Each item in the sidebar has a “Place” button (or double-click).
- Clicking it creates a new placement via POST API centered at `(stageWidth/2, stageHeight/2)`
  in canvas coordinates, then adds the Konva node immediately (optimistic UI).

**Zoom:**

- Zoom is a local React state (default 1.0, range 0.2 to 3.0).
- Mouse wheel zooms toward cursor position using Konva’s `scale` + `position` transforms.
- Zoom does not affect stored coordinates (always stored at scale 1.0).

## PDF Rendering Detail

pdfjs-dist runs entirely in the browser. Recommended approach:

```ts
// lib/renderPdf.ts (client-only utility)
import * as pdfjsLib from "pdfjs-dist";
pdfjsLib.GlobalWorkerOptions.workerSrc = "/pdf.worker.min.js"; // served from public/

export async function renderPdfPageToDataUrl(
  pdfUrl: string,
  pageNumber = 1,
  scale = 2.0   // render at 2x for sharpness, then CSS-scale down
): Promise<{ dataUrl: string; widthPx: number; heightPx: number }> {
  const pdf = await pdfjsLib.getDocument(pdfUrl).promise;
  const page = await pdf.getPage(pageNumber);
  const viewport = page.getViewport({ scale });
  const canvas = document.createElement("canvas");
  canvas.width = viewport.width;
  canvas.height = viewport.height;
  await page.render({ canvasContext: canvas.getContext("2d")!, viewport }).promise;
  return {
    dataUrl: canvas.toDataURL("image/png"),
    widthPx: viewport.width,
    heightPx: viewport.height,
  };
}
```

The `pdf.worker.min.js` file must be copied from `node_modules/pdfjs-dist/build/` into
`public/` during the Docker build (or via a Next.js custom webpack config).

## Scaling Math

All scaling logic lives in `src/lib/calibration.ts`:

```ts
export function pixelDistance(x1: number, y1: number, x2: number, y2: number): number {
  return Math.sqrt((x2 - x1) ** 2 + (y2 - y1) ** 2);
}

export function computePixelsPerCm(
  x1: number, y1: number,
  x2: number, y2: number,
  realCm: number
): number {
  return pixelDistance(x1, y1, x2, y2) / realCm;
}

export function cmToCanvasPx(cm: number, pixelsPerCm: number): number {
  return cm * pixelsPerCm;
}
```

**Important:** `pixelsPerCm` is computed once from the PDF rendered at a specific scale
factor (e.g. 2.0 via pdfjs). This scale factor must be consistent between calibration
and all subsequent editor sessions. Store the render scale in the project row or hardcode
it as a constant. The simplest approach is to always render at `scale = 1.0` during
calibration and all editor loads – this keeps pixel coordinates unambiguous.

## Key Implementation Notes

### 1. DB singleton

Next.js runs in a Node.js process but module cache is shared. Use a module-level
singleton to avoid opening multiple DB connections:

```ts
// src/db/index.ts
import Database from "better-sqlite3";
import { drizzle } from "drizzle-orm/better-sqlite3";
import * as schema from "./schema";

const sqlite = new Database(process.env.DATABASE_URL!.replace("file:", ""));
export const db = drizzle(sqlite, { schema });
```

### 2. Serving the PDF

The PDF must be served through an API route (not as a static file) because it lives in
`/data/uploads/` which is outside `public/`. The route at `GET /api/pdf/[projectId]`
reads the file with `fs.readFile` and returns it with `Content-Type: application/pdf`.
pdfjs-dist accepts a URL string, so pass `/api/pdf/[projectId]` as the document URL.

### 3. Placement persistence strategy

Do NOT auto-save on every mouse-move event. Persist on `dragend` only (Konva fires this
once when the user releases the piece). Debounce is optional but recommended (200ms) in
case of rapid drags. Rotation changes persist immediately on button click.

On page load, fetch `GET /api/layouts/[layoutId]` which returns the layout with all
placements joined to their furniture item (name, widthCm, heightCm, colour). Reconstruct
all Konva nodes from this data.

### 4. Deleting a furniture item from the library

If a furniture item is deleted while placements exist in any layout, the DB cascades the
deletion (see schema). On the frontend, re-fetching or receiving a 404 on a placement’s
furniture item should gracefully show a “missing item” placeholder or simply remove the
node.

### 5. Multiple layouts sharing the same furniture library

Layouts do NOT copy furniture items – they reference them by ID. If an item’s dimensions
are updated in the library, all layouts reflect the new dimensions on next load. This is
intentional and should be documented in the UI (“Editing a furniture item updates it in
all layouts.”).

### 6. Canvas coordinate system

Konva’s default coordinate origin is top-left. Rotations are clockwise. The `rotation`
property on a Konva `Group` or `Rect` is in degrees. When using a `Transformer`, rotations
are continuous (not snapped). The “Rotate 90°” button adds 90 to the stored value and
calls the API – it does not use the Transformer’s rotation handle.

### 7. Zoom implementation

```ts
const handleWheel = (e: Konva.KonvaEventObject<WheelEvent>) => {
  e.evt.preventDefault();
  const stage = stageRef.current!;
  const oldScale = stage.scaleX();
  const pointer = stage.getPointerPosition()!;
  const mousePointTo = {
    x: (pointer.x - stage.x()) / oldScale,
    y: (pointer.y - stage.y()) / oldScale,
  };
  const direction = e.evt.deltaY > 0 ? -1 : 1;
  const newScale = Math.max(0.2, Math.min(3, oldScale + direction * 0.05));
  stage.scale({ x: newScale, y: newScale });
  stage.position({
    x: pointer.x - mousePointTo.x * newScale,
    y: pointer.y - mousePointTo.y * newScale,
  });
};
```

### 8. Calibration UX state machine

The `useCalibration` hook manages three states:

- `idle` – waiting for first click
- `first_point` – first point placed, waiting for second click
- `done` – two points placed, modal open for cm input

On modal confirm, dispatch to the API. On cancel, reset to `idle`.

## Out of Scope (for this spec)

- Multiple PDF pages (only page 1 is used)

- Undo/redo
- Snapping to grid
- Collision detection between furniture pieces

## Acceptance Criteria

- [ ] A new project can be created with a name.
- [ ] A PDF can be uploaded and is persisted across container restarts.
- [ ] The calibration flow allows drawing a line on the PDF and entering a real-world cm value; `pixelsPerCm` is saved.
- [ ] Furniture items can be created, edited, and deleted in the library.
- [ ] Multiple layout variants can be created under one project.
- [ ] The canvas editor shows the floor plan PDF as a background.
- [ ] Furniture items from the library can be placed onto the canvas.
- [ ] Placed items render at the correct physical scale relative to the calibrated floor plan.
- [ ] Dragging a placed item persists its new position after `dragend`.
- [ ] Placed items can be rotated in 90° increments and removed.
- [ ] Two different layouts show independent furniture positions for the same set of items.
- [ ] Editing a furniture item’s dimensions in the library is reflected in all layouts on next load.
- [ ] The entire app runs as a single `docker compose up` command on port 3000.
- [ ] Data (DB + uploaded PDF) survives container restarts via the mounted volume.
